# Requirements Document: Agentic Upgrade Phase

## Introduction

This document specifies requirements for upgrading the Hospital Pre-Appointment Management System from a linear pipeline to a true agentic workflow. The system currently has deterministic prep generation, basic LangGraph workflow, and SQLite persistence. This upgrade adds voice input, EHR/FHIR integration, missing field detection, hospital lookup, enhanced scheduling, and conditional LangGraph flow.

## Glossary

- **Agent**: The LangGraph-based orchestration system that processes patient intake through multiple conditional nodes
- **Voice_Input_System**: Web Speech API integration for voice-to-text transcription
- **EHR_Service**: Electronic Health Record integration layer using FHIR protocol
- **FHIR_Client**: Client for fetching patient data from FHIR servers (HAPI FHIR test server)
- **FHIR_Normalizer**: Component that converts FHIR resources to internal intake schema
- **Missing_Field_Detector**: Python function that identifies required fields absent from intake data
- **Hospital_Lookup_Service**: Service using Overpass API and Nominatim for hospital search
- **Scheduling_Orchestrator**: Component that integrates with Google Calendar API for appointment booking
- **RulesEngine**: Deterministic medical preparation rules engine (source of truth)
- **LangGraph_Flow**: Conditional state machine workflow with nodes and edges
- **Mock_Mode**: System operation without external API credentials using simulated data
- **Intake_Data**: Structured patient information including symptoms, medications, allergies, demographics
- **Prep_Summary**: Patient-facing preparation instructions generated by the system
- **Clinical_Briefing**: Clinician-facing summary with EHR context and risk assessment

## Requirements

### Requirement 1: Voice Input Integration

**User Story:** As a patient, I want to provide my symptoms and medical history via voice input, so that I can complete intake faster and more naturally.

#### Acceptance Criteria

1. WHEN the patient clicks the microphone button, THE Voice_Input_System SHALL request microphone permission from the browser
2. WHEN microphone permission is granted, THE Voice_Input_System SHALL start recording audio using Web Speech API
3. WHILE recording is active, THE Voice_Input_System SHALL display a visual indicator showing recording status
4. WHEN the patient clicks the microphone button again, THE Voice_Input_System SHALL stop recording and send audio to transcription endpoint
5. WHEN audio is received at `/api/transcribe`, THE Agent SHALL transcribe speech to text using speech recognition library
6. WHEN transcription completes successfully, THE Agent SHALL populate the conversational query field with transcribed text
7. IF transcription fails, THEN THE Agent SHALL display an error message without blocking manual text entry
8. THE Voice_Input_System SHALL work in both HTTPS and localhost environments
9. IF the browser does not support Web Speech API, THEN THE Voice_Input_System SHALL disable the microphone button and show a tooltip
10. FOR ALL voice inputs, the system SHALL preserve the original transcript for audit purposes

### Requirement 2: EHR/FHIR Integration

**User Story:** As a clinician, I want patient data automatically pulled from the EHR system, so that intake is pre-filled with accurate medical history.

#### Acceptance Criteria

1. WHEN a patient ID is provided, THE FHIR_Client SHALL fetch Patient resource from HAPI FHIR test server
2. WHEN Patient resource is fetched, THE FHIR_Normalizer SHALL extract patient name, age group, and contact information
3. WHEN patient allergies are needed, THE FHIR_Client SHALL fetch AllergyIntolerance resources for the patient
4. WHEN AllergyIntolerance resources are fetched, THE FHIR_Normalizer SHALL convert them to a list of allergy names
5. WHEN patient medications are needed, THE FHIR_Client SHALL fetch active MedicationStatement resources
6. WHEN MedicationStatement resources are fetched, THE FHIR_Normalizer SHALL extract medication names and dosages
7. WHEN patient procedures are needed, THE FHIR_Client SHALL fetch completed Procedure resources
8. WHEN Procedure resources are fetched, THE FHIR_Normalizer SHALL map them to prior conditions list
9. WHEN patient vitals are needed, THE FHIR_Client SHALL fetch Observation resources with category vital-signs
10. WHEN Observation resources are fetched, THE FHIR_Normalizer SHALL extract vital sign values with units
11. THE EHR_Service SHALL merge FHIR data with manually entered intake data, preferring FHIR for structured fields
12. WHERE mock mode is enabled, THE FHIR_Client SHALL return realistic mock data without making API calls
13. IF FHIR server is unavailable, THEN THE Agent SHALL continue with manually entered data and log the error
14. THE FHIR_Normalizer SHALL produce intake data compatible with existing RulesEngine schema
15. FOR ALL FHIR data fetches, the system SHALL include a round-trip property test (fetch → normalize → validate schema)

---

### Requirement 3: Missing Field Detection

**User Story:** As a system, I want to detect missing required fields in patient intake, so that I can prompt for complete information before generating prep instructions.

#### Acceptance Criteria

1. WHEN intake data is received, THE Missing_Field_Detector SHALL identify all required fields for the appointment type
2. WHEN a required field is missing, THE Missing_Field_Detector SHALL add it to the missing fields list
3. WHEN chief complaint is missing, THE Missing_Field_Detector SHALL mark it as high priority
4. WHEN appointment type is missing, THE Missing_Field_Detector SHALL mark it as high priority
5. WHEN optional fields are missing, THE Missing_Field_Detector SHALL not block workflow progression
6. THE Missing_Field_Detector SHALL calculate a confidence score based on completeness (0.0 to 1.0)
7. IF confidence score is below 0.8, THEN THE Agent SHALL route to clarification node
8. WHEN clarification is needed, THE Agent SHALL generate intelligent follow-up questions for missing fields
9. WHEN follow-up questions are presented, THE Agent SHALL suggest common options for each field
10. THE Missing_Field_Detector SHALL update state with missing_fields list and confidence_score
11. FOR ALL intake data, the detector SHALL verify that completeness calculation is idempotent (running twice produces same result)

---

### Requirement 4: Hospital and Doctor Lookup

**User Story:** As a patient, I want to see nearby hospitals and doctors suitable for my procedure, so that I can choose a convenient and highly-rated facility.

#### Acceptance Criteria

1. WHEN procedure type is determined, THE Hospital_Lookup_Service SHALL query Overpass API for nearby hospitals
2. WHEN hospitals are found, THE Hospital_Lookup_Service SHALL fetch ratings and details from Nominatim
3. WHEN hospital data is retrieved, THE Agent SHALL filter hospitals by procedure capability
4. WHEN multiple hospitals match, THE Agent SHALL rank them by rating and distance
5. THE Hospital_Lookup_Service SHALL return hospital name, location, rating, and distance for each result
6. WHEN doctor suggestions are needed, THE Agent SHALL match specialty to procedure type
7. WHEN specialty is matched, THE Agent SHALL return a list of doctors with ratings and availability
8. WHERE mock mode is enabled, THE Hospital_Lookup_Service SHALL return realistic mock hospital data
9. IF Overpass API is unavailable, THEN THE Agent SHALL use fallback mock data and log the error
10. THE Agent SHALL display hospital suggestions in the frontend with ratings and location
11. FOR ALL hospital lookups, the system SHALL verify that distance calculations are consistent (same input produces same output)

---

### Requirement 5: Enhanced Scheduling Orchestration

**User Story:** As a patient, I want the system to automatically book my appointment and add it to my calendar, so that I don't have to manually coordinate scheduling.

#### Acceptance Criteria

1. WHEN a patient selects a time slot, THE Scheduling_Orchestrator SHALL validate slot availability
2. WHEN slot is available, THE Scheduling_Orchestrator SHALL create a Google Calendar event using calendar_service
3. WHEN calendar event is created, THE Scheduling_Orchestrator SHALL store the event ID in state
4. WHEN booking is confirmed, THE Scheduling_Orchestrator SHALL send confirmation via email or SMS
5. THE Scheduling_Orchestrator SHALL include appointment details, doctor name, and location in confirmation
6. WHEN appointment is booked, THE Agent SHALL generate patient prep summary using RulesEngine
7. WHEN prep summary is generated, THE Agent SHALL attach it to the calendar event as a note
8. WHERE mock mode is enabled, THE Scheduling_Orchestrator SHALL simulate booking without creating real calendar events
9. IF Google Calendar API is unavailable, THEN THE Agent SHALL save booking details to database and notify user
10. THE Scheduling_Orchestrator SHALL support cancellation and rescheduling of appointments
11. FOR ALL booking operations, the system SHALL verify that event creation is idempotent (duplicate requests do not create multiple events)

---

### Requirement 6: LangGraph Flow Refactoring

**User Story:** As a developer, I want the agent workflow to use conditional routing instead of linear pipeline, so that the system can handle complex decision logic and missing information gracefully.

#### Acceptance Criteria

1. THE LangGraph_Flow SHALL replace the linear pipeline with conditional nodes and edges
2. WHEN intake is received, THE Agent SHALL route to voice_input node if input mode is voice
3. WHEN voice input is processed, THE Agent SHALL route to conversation_intake node
4. WHEN conversation intake is complete, THE Agent SHALL route to missing_info_detector node
5. IF confidence score is below 0.8, THEN THE Agent SHALL route to clarification_agent node
6. IF confidence score is 0.8 or higher, THEN THE Agent SHALL route to process_intake node
7. WHEN clarification is needed, THE Agent SHALL end the current turn and wait for user response
8. WHEN intake is processed, THE Agent SHALL route to triage node
9. WHEN triage is complete, THE Agent SHALL route to protocol_retrieval node
10. WHEN protocols are retrieved, THE Agent SHALL route to admin_prep node
11. WHEN admin prep is complete, THE Agent SHALL route to hospital_suggestion node
12. WHEN hospital suggestions are ready, THE Agent SHALL route to scheduling_orchestrator node
13. WHEN scheduling is complete, THE Agent SHALL route to clinical_briefing node
14. WHEN clinical briefing is complete, THE Agent SHALL route to patient_message node
15. WHEN patient message is generated, THE Agent SHALL route to clinician_summary node
16. WHEN clinician summary is complete, THE Agent SHALL route to save node
17. THE LangGraph_Flow SHALL log each node transition to reasoning trace
18. THE Agent SHALL update state fields at each node (intake_data, triage_data, admin_prep_data, etc.)
19. IF any node encounters an error, THEN THE Agent SHALL add error to state and continue to next node
20. THE LangGraph_Flow SHALL support visualization using Mermaid diagram format
21. FOR ALL workflow executions, the system SHALL verify that state transitions are deterministic (same input produces same node sequence)

---

### Requirement 7: Agent State Extensions

**User Story:** As a developer, I want the agent state to include new fields for voice input, EHR data, and hospital suggestions, so that all workflow information is tracked in state.

#### Acceptance Criteria

1. THE Agent SHALL add conversation_data field to AgentState for voice and missing field tracking
2. THE conversation_data field SHALL include missing_fields list, confidence_score, and current_transcript
3. THE Agent SHALL add hospital_data field to AgentState for hospital lookup results
4. THE hospital_data field SHALL include hospital name, location, rating, and distance
5. THE Agent SHALL add ehr_context field to AgentState for FHIR data
6. THE ehr_context field SHALL include patient demographics, allergies, medications, and vitals
7. THE Agent SHALL preserve existing state fields for backward compatibility
8. THE Agent SHALL initialize all new fields in create_initial_state function
9. THE Agent SHALL update conversation_data when missing fields are detected
10. THE Agent SHALL update hospital_data when hospital lookup completes
11. FOR ALL state updates, the system SHALL verify that state schema is valid (all required fields present)

---

### Requirement 8: Frontend Voice Input UI

**User Story:** As a patient, I want a clear and intuitive voice input interface, so that I know when the system is listening and can easily provide voice input.

#### Acceptance Criteria

1. THE Frontend SHALL display a microphone button next to the conversational query textarea
2. WHEN the microphone button is clicked, THE Frontend SHALL change button color to indicate recording
3. WHILE recording is active, THE Frontend SHALL display a pulsing red dot with "Listening..." text
4. WHEN recording stops, THE Frontend SHALL display "Transcribing..." text
5. WHEN transcription completes, THE Frontend SHALL populate the textarea with transcribed text
6. THE Frontend SHALL allow patients to edit transcribed text before submission
7. THE Frontend SHALL support both voice input and manual text entry in the same session
8. IF microphone permission is denied, THEN THE Frontend SHALL display an error message with instructions
9. THE Frontend SHALL disable the microphone button while transcription is in progress
10. THE Frontend SHALL provide visual feedback for all voice input states (idle, recording, transcribing, complete, error)

---

### Requirement 9: EHR Tool Integration

**User Story:** As a developer, I want EHR services integrated into agent tools, so that the agent can fetch patient data during workflow execution.

#### Acceptance Criteria

1. THE Agent SHALL add ehr_enrichment_tool to agent/tools.py
2. WHEN ehr_enrichment_tool is called, THE Agent SHALL fetch patient data using EHR_Service
3. WHEN patient data is fetched, THE Agent SHALL merge it with existing intake_data
4. THE ehr_enrichment_tool SHALL update state with enriched intake_data and ehr_context
5. THE Agent SHALL add ehr_enrichment node to LangGraph_Flow
6. THE Agent SHALL route to ehr_enrichment node after intake processing if patient ID is available
7. WHERE mock mode is enabled, THE ehr_enrichment_tool SHALL use mock FHIR data
8. IF EHR fetch fails, THEN THE ehr_enrichment_tool SHALL log error and continue with existing data
9. THE ehr_enrichment_tool SHALL preserve manually entered data when merging with EHR data
10. THE Agent SHALL log EHR enrichment status to reasoning trace

---

### Requirement 10: Hospital Lookup Tool Integration

**User Story:** As a developer, I want hospital lookup integrated into agent tools, so that the agent can suggest hospitals during workflow execution.

#### Acceptance Criteria

1. THE Agent SHALL add hospital_lookup_tool to agent/tools.py
2. WHEN hospital_lookup_tool is called, THE Agent SHALL query Hospital_Lookup_Service with procedure type
3. WHEN hospitals are found, THE Agent SHALL update state with hospital_data
4. THE hospital_lookup_tool SHALL rank hospitals by rating and distance
5. THE Agent SHALL add hospital_suggestion node to LangGraph_Flow
6. THE Agent SHALL route to hospital_suggestion node after admin_prep
7. WHERE mock mode is enabled, THE hospital_lookup_tool SHALL return mock hospital data
8. IF hospital lookup fails, THEN THE hospital_lookup_tool SHALL log error and continue without suggestions
9. THE hospital_lookup_tool SHALL support filtering hospitals by procedure capability
10. THE Agent SHALL log hospital lookup results to reasoning trace

---

### Requirement 11: Mock Mode Preservation

**User Story:** As a developer, I want the system to work in mock mode without external API credentials, so that development and testing can proceed without dependencies.

#### Acceptance Criteria

1. THE Agent SHALL support mock_mode flag for all external services
2. WHEN mock_mode is enabled, THE FHIR_Client SHALL return realistic mock patient data
3. WHEN mock_mode is enabled, THE Hospital_Lookup_Service SHALL return mock hospital data
4. WHEN mock_mode is enabled, THE Scheduling_Orchestrator SHALL simulate booking without API calls
5. WHEN mock_mode is enabled, THE Voice_Input_System SHALL work normally (Web Speech API is free)
6. THE Agent SHALL default to mock_mode=True for all services
7. THE Agent SHALL allow mock_mode to be disabled via environment variables
8. WHERE mock_mode is enabled, THE Agent SHALL log that mock data is being used
9. THE Agent SHALL ensure mock data is realistic and covers common scenarios
10. FOR ALL mock data, the system SHALL verify that it matches the schema of real API responses

---

### Requirement 12: Deterministic Rules Preservation

**User Story:** As a clinician, I want the system to continue using deterministic RulesEngine for medical instructions, so that prep quality and safety are maintained.

#### Acceptance Criteria

1. THE Agent SHALL continue using RulesEngine as the source of truth for medical preparation rules
2. WHEN prep instructions are generated, THE Agent SHALL apply RulesEngine rules based on appointment type and procedure
3. THE Agent SHALL not allow LLM to modify medical instructions, fasting hours, or arrival times
4. THE Agent SHALL use LLM only for phrasing and tone enhancement of prep messages
5. THE RulesEngine SHALL remain unchanged by the agentic upgrade
6. THE Agent SHALL preserve all existing deterministic rules (surgery, endoscopy, imaging, lab work)
7. THE Agent SHALL ensure prep summary quality does not degrade with new features
8. THE Agent SHALL validate that RulesEngine output is used in final prep message
9. THE Agent SHALL log which rules were applied to reasoning trace
10. FOR ALL prep generation, the system SHALL verify that RulesEngine output is deterministic (same input produces same rules)

---

### Requirement 13: API Route Extensions

**User Story:** As a frontend developer, I want new API routes for voice transcription and hospital lookup, so that the frontend can access new agentic features.

#### Acceptance Criteria

1. THE Agent SHALL add POST `/api/transcribe` route to app.py for voice transcription
2. WHEN `/api/transcribe` receives audio file, THE Agent SHALL transcribe it using speech recognition
3. WHEN transcription succeeds, THE Agent SHALL return JSON with transcribed text
4. IF transcription fails, THEN THE Agent SHALL return error JSON with message
5. THE Agent SHALL add POST `/api/hospital-lookup` route to app.py for hospital search
6. WHEN `/api/hospital-lookup` receives procedure type, THE Agent SHALL query Hospital_Lookup_Service
7. WHEN hospitals are found, THE Agent SHALL return JSON with hospital list
8. THE Agent SHALL add POST `/api/analyze` route for step 2 of wizard (triage and doctor suggestions)
9. WHEN `/api/analyze` receives intake data, THE Agent SHALL perform triage and return doctor suggestions
10. THE Agent SHALL add POST `/api/book-appointment` route for step 4 of wizard (booking confirmation)
11. WHEN `/api/book-appointment` receives booking details, THE Agent SHALL run full agent workflow and return prep summary
12. THE Agent SHALL handle errors gracefully for all new routes and return appropriate HTTP status codes
13. THE Agent SHALL log all API requests and responses for debugging
14. FOR ALL API routes, the system SHALL verify that error handling is consistent (all errors return JSON with error field)

---

### Requirement 14: Frontend Wizard Flow

**User Story:** As a patient, I want a seamless multi-step wizard interface, so that I can complete intake, review doctor suggestions, and confirm booking in a guided flow.

#### Acceptance Criteria

1. THE Frontend SHALL implement a 5-step wizard: Intake → Analysis → Confirm → Booking → Results
2. WHEN step 1 is complete, THE Frontend SHALL call `/api/analyze` and display triage results
3. WHEN step 2 is complete, THE Frontend SHALL display doctor cards with ratings and available slots
4. WHEN a doctor and slot are selected, THE Frontend SHALL enable the "Confirm Selection" button
5. WHEN step 3 is complete, THE Frontend SHALL display booking confirmation details
6. WHEN "Confirm & Book" is clicked, THE Frontend SHALL call `/api/book-appointment` and show loading state
7. WHEN booking completes, THE Frontend SHALL display results with patient prep and clinician summary
8. THE Frontend SHALL display urgency tag (routine, urgent, emergency) based on triage results
9. THE Frontend SHALL allow patients to go back to previous steps to modify selections
10. THE Frontend SHALL preserve form data when navigating between steps
11. THE Frontend SHALL display error messages inline without blocking the entire workflow
12. THE Frontend SHALL show a progress indicator in the sidebar with completed steps marked
13. FOR ALL wizard steps, the system SHALL verify that navigation is consistent (back button returns to previous step)

---

### Requirement 15: Correctness Properties

**User Story:** As a developer, I want property-based tests for critical system components, so that I can verify correctness across a wide range of inputs.

#### Acceptance Criteria

1. THE System SHALL include a round-trip property test for FHIR normalization (fetch → normalize → validate schema)
2. THE System SHALL include an idempotence property test for missing field detection (running twice produces same result)
3. THE System SHALL include a consistency property test for hospital distance calculations (same input produces same output)
4. THE System SHALL include an idempotence property test for calendar event creation (duplicate requests do not create multiple events)
5. THE System SHALL include a determinism property test for LangGraph state transitions (same input produces same node sequence)
6. THE System SHALL include a schema validation property test for agent state updates (all required fields present)
7. THE System SHALL include a determinism property test for RulesEngine output (same input produces same rules)
8. THE System SHALL include a consistency property test for API error handling (all errors return JSON with error field)
9. THE System SHALL include a consistency property test for wizard navigation (back button returns to previous step)
10. THE System SHALL include a round-trip property test for voice transcription (record → transcribe → validate text format)
